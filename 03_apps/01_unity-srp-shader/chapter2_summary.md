# 02. シングルPassフォワードレンダリング

Universal Render Pipeline (URP) における主要なレンダリング方式である「シングルPassフォワードレンダリング」について、従来の方式との比較を交えて整理する。

## 1. 従来のフォワードレンダリング (Built-in)

従来の「フォワードレンダリング」方式では、ライトの影響を計算するために複数のPass（描画工程）を重ねる必要があった。

* **処理の仕組み**: 
    - メインライトの描画を行う「Base Pass」と、それ以外の追加ライトを計算する「Additional Pass」に分かれている。
* **メリット**:
    - 処理負荷をPass単位で分散できる。
    - 理論上、ライトの個数に上限がない。
* **デメリット**:
    - **ドローコールの増大**: ライトが1つ増えるごとにPassが追加されるため、ライトの数に比例してドローコールが増大し、CPU/GPUコストが非常に高くなる。

---

## 2. シングルPassフォワードレンダリング (URP)

URPが採用している改良版の方式。その名の通り「1回のPass」で全ての計算を完結させる。

* **処理の仕組み**:
    - 1つのPassの中で、メインライトおよび全ての追加ライトの描画計算を同時に行う。
* **メリット**:
    - **ドローコールの最適化**: ライトの個数が増えてもPassが1つで済むため、ドローコールの回数が劇的に抑えられ、パフォーマンスが向上する。
* **デメリット / 制約**:
    - **ライトの個数制限**: 1度の計算で処理できるデータ量に限りがあるため、ライトの同時適用数に上限が存在する（1オブジェクトあたり最大8個）。

---

## 📝 比較まとめ

| 特徴 | フォワード (旧) | シングルPassフォワード (URP) |
| :--- | :--- | :--- |
| **Pass数** | ライト数に応じて増加 | **常に1回** |
| **ドローコール** | 多い（ライト数に依存） | **少ない（ライト数に依存しない）** |
| **ライト上限** | 無制限 | 最大8個まで |
| **主な用途** | 複雑なライティング | モバイル・VRを含む汎用的な最適化 |
