# 01. スクリプタブルレンダーパイプライン (SRP) 概論

## 🗝 基礎用語の定義

* **ドローコール (Draw Call)**
    - CPUからGPUに対する「描画命令」のこと。
    - 描画に必要な頂点情報やテクスチャなどのリソースデータをグラフィックメモリへ送信し、描画を指示する一連のプロセス。
    - **性質**: 発行の際、CPUとGPUの間で同期が必要となるため、回数が増えるほど処理負荷が高まる。

* **レンダーパイプライン (Render Pipeline)**
    - リソースの管理とドローコールの発行を制御するメカニズム。
    - 描画の速度（パフォーマンス）と見た目（グラフィックス品質）を決定づける心臓部にあたる。

---

## 🔄 組み込み (Built-in) から SRP への進化

これまでのUnityでは「組み込みレンダーパイプライン」が主流であったが、現在はより自由度の高い **SRP (Scriptable Render Pipeline)** への移行が進んでいる。

### 比較構造

| 項目 | 組み込み (Built-in) | SRP (URP / HDRP) |
| :--- | :--- | :--- |
| **制御の仕組み** | C++ (Unityエンジンコアと一体) | C# (スクリプトによる制御) |
| **カスタマイズ性** | 低い（内部がブラックボックス） | 高い（モジュール単位で拡張可能） |
| **提供形態** | Unity本体に内蔵 | Package Managerによる提供 |

### 組み込み型の主な課題（廃止の背景）
1. **カスタマイズの制限**: エンジンのコア実装に密結合しているため、ユーザー側での最適化や修正が困難。
2. **保守の複雑化**: あらゆるプラットフォームに対応する万能型ゆえに、コードが巨大かつ複雑でメンテナンスコストが高い。
3. **更新の独立性**: 新しい技術を導入する際、Unity本体のバージョンアップを待つ必要があり、柔軟な進化が難しかった。

---

## 🚀 SRP (Scriptable Render Pipeline) の本質

SRPとは、**「プログラミング可能なレンダーパイプライン」**である。描画処理をC#レイヤーから呼び出せるように設計されている。



### 独立したパッケージ管理
SRPは「Package Manager」を介して提供されるため、Unity本体のバージョンとは切り離して管理・更新が可能。プロジェクトのニーズに合わせて必要なライブラリを選択・追加できる。

### 公式提供のテンプレート
Unityは、低レイヤーの「SRCCore」をベースとした2種類の公式パイプラインを提供している。
* **Universal RP (URP)**: 幅広いプラットフォームに対応する汎用型。軽量かつ高性能で、従来のパイプラインの正当な後継。
* **HDRP**: ハイエンド環境（PC/コンシューマ機）に特化した高精度型。計算負荷は高いが、フォトリアルな表現に適している。

---

## 📝 まとめ
* レンダーパイプラインは、アプリケーションの実行速度と視覚的クオリティを左右する最重要要素である。
* SRPの登場により、エンジンのコア実装と描画ロジックが**疎結合**となり、プロジェクトごとに最適化された描画環境を構築できるようになった。
